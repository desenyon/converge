# Converge Architecture

Converge is organized around one rule: every operation is scoped to a specific target repository.

## 1. Project Context
- `ProjectContext` resolves the repository root from the user-provided path.
- It owns all derived artifact locations, including `.converge/graph.db`, validation sandboxes, and the default `.venv`.
- CLI commands should use this context immediately on entry and pass it down instead of relying on process-global state.

## 2. Scanning Pipeline
- `ProjectParser` reads declared dependencies from `pyproject.toml` and `requirements*.txt`.
- `PythonASTParser` scans Python modules, records external imports, and skips internal modules discovered in the repo root or `src/`.
- `ServiceDetector` adds route entities derived from framework decorators.
- `Scanner` combines these stages into one deterministic graph snapshot for the target repository.

## 3. Graph Persistence
- `GraphStore` persists the graph into a SQLite database at `.converge/graph.db` inside the target repository.
- The database is a derived artifact and can be regenerated by rerunning `scan`.
- `networkx` is used for in-memory analysis after the graph is loaded from SQLite.

## 4. Diagnostics and Repair Planning
- `ConflictDetector` finds unresolved imports, unused dependencies, and version-clash edges.
- `RepairPlanner` converts those conflicts into explicit repair actions.
- `fix --apply` only writes back to the repo after validation has selected a passing plan.

## 5. Environment and Validation
- `EnvironmentManager` provisions the repo-local environment at `.venv`.
- `create` installs the package set derived from the target repository graph, not from any global database.
- Validation uses an isolated sandbox and smoke-import checks before manifest mutations are applied to the host repository.
